# 城市道路既有通信光纤 DAS 空洞探测场景描述与技术路线规划

## 1. 场景定位

本项目面向城市道路地下空洞与浅层病害探测场景。典型对象包括道路下方空洞、脱空、松散回填区、废弃管沟、管线周边扰动、弱夹层和局部破碎带等浅层异常体。

项目假设道路一侧或道路附近已经存在沿道路方向延伸的通信光纤。该光纤可接入 DAS 设备，形成近似连续的线性接收阵列。震源采用锤击、落锤、小型主动源或车辆激发等浅层震源。通过沿道路方向布设多个激发点，采集多炮 DAS-like 地震记录，分析瑞雷波直达波、异常体绕射/散射波、能量异常和相干性变化，进而研究道路下方疑似空洞或弱异常体的位置与埋深。

本项目定位为科研级算法原型平台，重点研究城市道路单侧既有通信光纤 DAS-like 接收条件下的正演模拟、绕射/散射识别、多炮联合定位、定位可信度评价和局部全波场验证。项目不追求工业级工程软件实现，也不将当前结果直接作为工程确诊结论，而是用于验证方法可行性、分析适用条件、沉淀算法流程和形成后续扩展基础。

---

## 2. 典型工程布设

### 2.1 道路与光纤布设

典型场景中，道路具有一定横向宽度，通信光纤通常位于道路一侧、人行道、绿化带、管廊附近或道路边缘。光纤沿道路方向延伸，可近似看作一条连续接收线。

震源线通常布设在道路另一侧、车道边缘或可施工区域内，沿道路方向进行多点激发。由于城市道路环境限制，通常难以在道路中间密集布设传统检波器，因此利用既有通信光纤作为 DAS 接收阵列具有工程现实意义。

需要注意的是，该观测方式属于单侧观测几何。光纤位于道路一侧，震源位于道路另一侧或道路边缘，异常体位于道路下方。在这种几何条件下，异常体沿道路方向的位置通常相对容易约束，而横向位置和埋深之间可能存在一定耦合。因此，后续定位结果必须同时给出误差评价、置信度指标和适用边界说明。

### 2.2 三维坐标系

本项目采用如下坐标约定：

```text
x：沿道路方向，也是光纤延伸方向；
y：横穿道路方向；
z：深度方向，向下为正。
```

典型几何关系为：

```text
DAS 光纤线：近似位于 y = 0；
道路区域：0 < y < W；
震源线：近似位于 y = W；
地下异常体：位于道路下方，坐标为 x0, y0, h。
```

其中：

```text
W：道路宽度，或震源线与光纤线之间的横向距离；
x0：异常体沿道路方向的位置；
y0：异常体横穿道路方向的位置；
h：异常体埋深。
```

为了保证项目长期可维护，所有正演、扫描、绘图、报告和测试流程应统一使用同一套几何配置。道路宽度、光纤位置、震源位置、通道间距、炮点间距、异常体位置等参数不应在不同模块中重复硬编码，避免出现同一场景在 geometry、forward、scan、wavefield、plot 等流程中参数不一致的问题。

## 3. 数据采集形式

项目主要考虑主动源 DAS-like 采集。数据结构可表示为：

```text
shot × time × channel
```

其中：

```text
shot：不同激发点；
time：时间采样；
channel：DAS 光纤上的空间通道。
```

核心输出数据建议包括：

```text
data：合成 DAS-like 记录，shape = n_shot, n_time, n_channel；
time_axis：时间轴；
channel_x：DAS 通道坐标；
shot_x：炮点坐标；
metadata：几何、速度、异常体、噪声、DAS-like 近似方式等元数据。
```

典型采集要素如下：

| 要素 | 描述 |
|---|---|
| 激发方式 | 锤击、落锤、小型可控震源、车辆激发 |
| 激发位置 | 沿道路方向多点激发，通常与光纤线近似平行 |
| 接收方式 | DAS 光纤沿道路方向连续接收 |
| 主要有效信号 | 瑞雷波直达波、异常体绕射波、散射波 |
| 主要干扰 | 交通噪声、路面耦合变化、弱耦合段、坏道、环境振动、强面波遮挡 |
| 数据目标 | 从多炮 DAS-like 记录中识别异常体诱导的绕射/散射特征 |

当前项目中的 DAS 结果应称为 DAS-like response approximation，即 DAS 类接收响应近似。除非完整考虑光纤方向、gauge length、耦合条件、解调过程和仪器响应，否则不能称为真实 DAS 仪器模拟。

## 4. 目标异常体类型

本项目关注的是浅层道路病害，不是深部油气地震目标。典型异常体包括：

| 异常类型 | 工程含义 | 可能波场响应 |
|---|---|---|
| 空洞 | 地下空腔、脱空、塌陷前隐患 | 强散射、绕射、局部能量异常 |
| 松散区 | 回填不密实、扰动土体 | 低速异常、相干性降低、能量衰减 |
| 管沟 | 管线沟槽、回填边界 | 线状或带状散射、边界响应 |
| 管线 | 圆柱状管体或管周扰动 | 局部散射、绕射，可能出现高/低速异常 |
| 弱夹层 | 局部软弱层或低速夹层 | 波速变化、能量衰减、相位扰动 |
| 破碎带 | 裂隙、扰动区、局部破碎土体 | 散射增强、相干性降低 |

在快速科研验证阶段，可以采用多个等效散射点表示异常体形状。例如，一个空洞可以由中心散射点或边界散射点集合近似，一个管沟可以由带状散射点集合近似。这种做法属于运动学等效散射近似，不应视为错误。后续可以逐步加入更严格的边界散射、有限差分、有限元、谱元、边界元和全波形弹性模拟方法，但不应破坏已有等效散射模型。

## 5. 核心物理机制

本项目关注的主要物理过程为：浅层主动源激发的弹性波场，在城市道路复杂近地表介质中传播，并在空洞、脱空、松散区、管沟、管线等异常体附近发生散射、绕射、反射、透射、模式转换和能量衰减。这些波场扰动被道路一侧既有通信光纤 DAS-like 接收，通过多炮联合分析实现异常体识别、定位和参数估计。

锤击、落锤、小型震源或车辆激发会在道路近地表产生以瑞雷波为主，同时包含 P 波、S 波、面波和导波的复杂弹性波场。波场在道路层状结构中传播时，会受到路面层、基层、路基、回填土和管线结构的影响，表现出速度变化、频散、能量衰减和相干性差异。

当波场遇到空洞、脱空或松散区等异常体时，会产生散射、绕射、反射、透射、模式转换和能量异常。这些异常响应会改变 DAS-like 记录中的走时、振幅、频谱、波形和相干性。

DAS 光纤不是传统点式检波器，它主要记录沿光纤方向、一定 gauge length 范围内的应变或应变率响应。因此，最终算法需要逐步考虑光纤方向、通道间距、gauge length、耦合状态和 DAS 对不同传播方向波场的敏感性。

真实异常体应在多炮记录中表现出几何一致性。通过多炮联合分析，可以利用不同震源、不同接收通道之间的走时、能量和相干性异常，对异常体的沿路位置、横向位置和埋深进行定位，并评估置信度和不确定性。

当前阶段可以先采用快速运动学与等效散射近似。典型散射走时可表示为：

```text
t_scatter = t0
          + distance(source, anomaly) / v_eff
          + distance(anomaly, receiver) / v_eff
```

直达瑞雷波走时可表示为：

```text
t_direct = t0 + distance(source, receiver) / v_rayleigh
```

其中，source 为震源点，receiver 为 DAS-like 接收通道，anomaly 为异常体中心或等效散射点，v_eff 为等效瑞雷波速度或分层等效速度。

## 6. 项目要解决的核心科学问题

本项目不是单纯生成波场图，而是围绕以下科学与工程问题展开：

| 核心科学问题 | 对应算法任务 |
|---|---|
| 道路浅层异常体是否能被 DAS 主动源波场识别 | 构建无异常体和含异常体的多炮正演记录，对比直达波、绕射波、散射波、局部能量和相干性差异 |
| 单侧 DAS 观测几何下能否实现可靠定位 | 建立道路-光纤-震源-异常体统一几何模型，实现 x-y-h 多炮联合扫描定位 |
| DAS 接收机制对异常识别有什么影响 | 从点式接收近似逐步扩展到 gauge length 轴向应变近似，分析接收方式对波形、能量和定位结果的影响 |
| 如何从复杂波场中提取稳定异常特征 | 实现直达波预测、直达波压制、局部能量、残差能量、频带能量、相干性和绕射聚焦属性 |
| 不同模型和算法的适用边界是什么 | 开展几何、速度、噪声、坏道、异常体深度和散射强度等敏感性实验，形成适用边界说明 |
| 定位结果如何评价可信度 | 建立 peak sharpness、score contrast、multi-shot consistency、y-depth coupling warning 和 low confidence flag 等评价指标 |
| 运动学近似与更真实波场之间差异多大 | 引入局部 elastic2d 或 simplified elastic3d 验证，对比等效散射结果与局部全波场响应 |
| 如何形成可持续扩展的科研级平台 | 建立统一配置、标准实验集、自动化报告、文档和测试体系，为后续 FEM、SEM、BEM、FWI refinement 预留接口 |

## 7. 项目技术主线

项目主线应始终围绕以下流程推进：

```text
道路地下模型
  -> 多炮正演数据
  -> DAS-like 接收响应
  -> 绕射/散射特征识别
  -> 多炮联合定位
  -> 定位误差与置信度评估
  -> 局部全波场验证
  -> 逐步反演改进
```

具体实现上，应遵循由简到繁、由可控到真实、由单一演示到系统实验的路线：

```text
统一几何与配置
  -> 运动学多炮正演
  -> 异常特征提取
  -> x-y-h 扫描定位
  -> 置信度与鲁棒性评价
  -> DAS-like 接收机制增强
  -> 道路分层模型增强
  -> 局部全波场验证
  -> 科研级图件、报告和实验体系沉淀
```

项目最终应形成一个稳定的科研级算法原型平台，而不是单个演示脚本。平台需要能够从配置生成模型、正演数据、提取特征、扫描定位、评价置信度、输出图件和报告，形成完整闭环。

## 8. 项目路线规划

建议将项目按照六轮推进。前四轮打通科研级主线，第五轮提高 DAS-like 接收和道路模型真实性，第六轮形成局部全波场验证和成果沉淀。该路线不要求工业级实现，但要求每一轮都有清晰的算法目标、可运行示例、测试和文档说明。

### 8.1 第一轮：场景建模与统一算法框架

第一轮的重点不是追求复杂算法，而是把城市道路、通信光纤、震源线、异常体、速度模型和数据结构统一起来，建立后续所有算法共享的基础框架。

这一轮主要解决：项目研究对象是什么、坐标系如何定义、几何参数如何统一、异常体如何描述、后续正演和定位如何使用同一套参数。

主要任务包括：

```text
1. 建立统一三维坐标系；
2. 建立道路、光纤、震源线和异常体的基础几何模型；
3. 建立统一配置系统，保证正演、扫描、绘图、报告都使用同一套几何参数；
4. 建立异常体数据结构，支持空洞、松散区、管沟、管线、弱夹层、破碎带等类型；
5. 建立速度模型基础接口，先支持 uniform effective Rayleigh velocity；
6. 建立 metadata 机制，记录每次实验的几何、速度、异常体、近似条件和算法版本；
7. 建立基础测试体系，保证几何距离、路径距离、参数同步和数据结构正确。
```

阶段产出：

```text
统一项目结构；
统一几何配置；
异常体基础描述；
速度模型基础接口；
metadata 记录机制；
基础测试体系。
```

阶段验收标准：

```text
能够用一套配置生成道路、光纤、震源点、接收通道和异常体坐标；
能够计算 source-receiver 距离和 source-anomaly-receiver 绕射路径；
修改道路宽度、通道间距、炮点数量或异常体位置后，所有模块同步变化。
```

### 8.2 第二轮：DAS-like 多炮运动学正演

第二轮开始生成合成数据。目标是用快速、可解释的运动学和等效散射近似，生成多炮 DAS-like 记录，为后续异常体识别和定位提供可控数据。

这一轮不追求完整弹性波模拟，而是先回答一个核心问题：在理想化条件下，道路浅层异常体是否能在单侧 DAS-like 多炮记录中形成可识别的绕射或散射响应。

主要任务包括：

```text
1. 实现基础震源子波，例如 Ricker 子波；
2. 实现瑞雷波直达波走时和波形模拟；
3. 实现异常体等效散射和绕射响应；
4. 支持一个异常体由多个散射点近似表达；
5. 支持多炮、多通道数据生成，输出 shot × time × channel 数据；
6. 支持几何扩散、简单吸收衰减、散射强度控制和随机噪声；
7. 实现背景模型和异常体模型对比；
8. 输出炮集图、波形图和基础 QC 图件。
```

阶段产出：

```text
多炮 DAS-like 合成数据；
直达波模拟；
异常体绕射/散射模拟；
背景模型与异常模型对比；
基础炮集图和 QC 图件。
```

阶段验收标准：

```text
能够生成无异常体和含异常体的多炮记录；
异常体位置、埋深、散射强度改变后，合成记录中的绕射响应随之变化；
固定随机种子后结果可复现；
所有输出都明确标注 kinematic approximation 和 DAS-like approximation。
```

### 8.3 第三轮：绕射/散射特征提取与多炮联合定位

第三轮是项目从能模拟走向能识别、能定位的关键阶段。目标是从多炮 DAS-like 数据中提取异常体相关特征，并通过多炮联合扫描得到异常体候选位置。

这一轮重点解决：异常体响应如何从强直达波、面波和噪声中分离出来；单侧 DAS 观测条件下，能否对异常体的 x 位置、横向位置 y 和埋深 h 进行联合估计。

主要任务包括：

```text
1. 实现直达波走时预测；
2. 实现直达波压制、静音或背景扣除；
3. 实现局部能量、残差能量、频带能量和相干性属性；
4. 实现候选异常体位置的理论绕射走时计算；
5. 实现 x-y-h 网格扫描；
6. 实现基于绕射能量聚焦、semblance 或 residual focusing 的评分方法；
7. 输出 score volume，而不是只输出单一最佳点；
8. 输出 best location、best score、定位误差和多维得分切片；
9. 支持多炮联合定位，并与单炮或少炮定位结果进行对比。
```

阶段产出：

```text
异常特征提取模块；
绕射走时预测模块；
x-y-h 多炮扫描定位模块；
score volume；
定位结果图；
真值对比和误差统计。
```

阶段验收标准：

```text
在简单单异常体合成模型中，扫描结果能够接近真值；
增加炮点数量后，定位稳定性提高；
降低信噪比或减少炮点后，得分体变模糊；
输出结果不仅包含 best location，还包含 score volume、误差和扫描参数。
```

### 8.4 第四轮：置信度评价、鲁棒性分析与适用边界

第四轮的目标是让项目从能给出一个位置进一步发展到能说明这个位置是否可信。单侧 DAS 观测几何天然存在非唯一性，尤其是 y 方向位置和埋深 h 之间容易耦合，因此必须建立可信度评价和失败场景说明。

主要任务包括：

```text
1. 实现 peak sharpness，用于评价得分峰是否清晰；
2. 实现 score contrast，用于评价最佳点与背景得分的差异；
3. 实现 multi-shot consistency，用于评价不同炮点对同一异常体位置的支持程度；
4. 实现 y-depth coupling warning，用于提示横向位置和埋深耦合；
5. 实现 low confidence flag，用于标记低可信定位结果；
6. 加入交通噪声、随机噪声、坏道、弱耦合段、速度误差等扰动；
7. 开展参数敏感性实验，包括炮点数量、通道间距、道路宽度、异常体埋深、异常体横向位置、速度误差和信噪比；
8. 自动生成 Markdown 置信度报告和适用边界说明。
```

阶段产出：

```text
定位置信度评价模块；
噪声和坏道扰动实验；
几何敏感性分析；
速度误差敏感性分析；
低置信度提示；
适用边界报告。
```

阶段验收标准：

```text
高信噪比、多炮、合理速度模型条件下，定位结果置信度较高；
低信噪比、少炮、速度误差较大或 y-h 耦合明显时，系统能够降低置信度并给出提示；
报告中明确说明低置信度结果不能作为工程确诊结论。
```

### 8.5 第五轮：DAS-like 接收机制与道路近地表模型增强

第五轮开始提高物理真实性，但仍然保持科研级、可控、可解释，不直接跳到复杂工业级模拟。重点是从简单点式接收和均匀速度模型，逐步过渡到更接近通信光纤 DAS 和道路近地表结构的模型。

这一轮解决的问题是：DAS-like 接收方式本身会不会改变异常体识别效果，道路分层结构和近地表速度变化会不会影响定位结果。

主要任务包括：

```text
1. 在点式接收近似基础上，实现 gauge length 轴向差分应变近似；
2. 支持 DAS-like 接收等级标记，例如 point receiver approximation、gauge-length strain approximation；
3. 加入光纤方向、通道间距、gauge length 对响应的影响；
4. 实现 layered-effective velocity model，用于描述道路面层、基层、路基和回填土等简化分层结构；
5. 加入简单频散或频带速度差异近似，用于描述瑞雷波在浅层结构中的速度变化；
6. 支持更丰富的异常体几何，例如椭圆空洞、矩形脱空、带状松散区、管沟和管线近似；
7. 对比点式接收与 gauge length 近似下的定位差异；
8. 对比均匀速度与分层等效速度下的定位偏差。
```

阶段产出：

```text
DAS-like gauge length 近似；
分层等效速度模型；
道路近地表简化模型；
复杂异常体几何近似；
接收机制对定位影响分析；
速度模型对定位影响分析。
```

阶段验收标准：

```text
gauge length 参数变化能够改变 DAS-like 响应；
分层速度模型能够影响绕射走时和定位结果；
复杂异常体几何能够通过等效散射点或简化边界表达；
原有运动学主线和扫描定位功能仍然保持可用。
```

### 8.6 第六轮：局部全波场验证、综合实验与科研成果沉淀

第六轮是科研级实现的收口阶段。目标不是把项目升级成工业级软件，而是在现有运动学扫描和 DAS-like 近似基础上，引入局部全波场验证和综合实验体系，说明算法在哪些条件下有效、在哪些条件下不足。

这一轮重点解决：运动学等效散射结果与更真实波场之间差异多大；最终项目如何形成可复现实验、可解释图件和科研报告。

主要任务包括：

```text
1. 预留并实现局部 elastic2d 或 simplified elastic3d 验证接口；
2. 在局部小模型中模拟空洞、松散区或管沟对波场的影响；
3. 对比运动学等效散射结果与局部全波场结果中的绕射、散射、能量异常和相干性变化；
4. 总结运动学扫描定位在不同场景中的偏差规律；
5. 建立标准实验集，包括理想单空洞、多异常体、低信噪比、坏道、速度误差、弱散射体、深浅不同异常体等；
6. 形成自动化实验报告流程；
7. 输出科研级图件，包括炮集对比图、能量属性图、扫描得分体切片、置信度图、误差统计图和失败案例图；
8. 编写项目总文档，明确当前已实现能力、近似条件、适用范围、失败场景和后续可扩展方向。
```

阶段产出：

```text
局部全波场验证接口；
标准实验集；
运动学与全波场对比；
算法适用性总结；

```

## 9. 项目边界与限制

本项目定位如下：

```text
面向城市道路空洞探测的科研级算法原型平台；
用于研究 DAS-like 多炮数据中的瑞雷波绕射/散射响应；
用于验证不同几何、速度、异常体、噪声条件下的定位可行性；
用于分析单侧 DAS 观测条件下异常体定位的误差、置信度和适用边界；
通过局部 elastic2d、simplified elastic3d、FEM、SEM、BEM、FWI refinement 等更高精度方法逐步增强物理真实性。
```

因此，所有输出图件、报告和文档必须诚实说明：

```text
当前采用的是运动学近似还是全波场模拟；
当前 DAS 是 DAS-like approximation 还是完整仪器模拟；
当前速度是 uniform、layered-effective 还是真实分层弹性模型；
当前定位结果是否存在低置信度、深度偏差或 y-depth 耦合；
当前结果能否作为科研分析依据，而不是工程确诊结论。
```

需要明确避免以下问题：

```text
不要把 DAS-like 近似说成完整 DAS 仪器模拟；
不要把运动学等效散射说成完整弹性波模拟；
不要把局部全波场验证说成完整工业级三维模拟平台；
不要只输出最佳定位点而不输出 score volume 和置信度；
不要忽略单侧观测几何下 y-depth 耦合的问题；
不要为了图件效果夸大定位精度；
不要破坏已有等效散射点模型，后续高级模型应作为增量扩展。
```

## 11. 可写入 README 或总计划的精简版

本项目面向城市道路地下空洞与浅层病害探测场景。典型布设为：道路一侧或附近存在沿道路方向延伸的通信光纤，可通过 DAS 设备形成近似连续接收阵列；道路另一侧或沿道路方向布设锤击、落锤、小型主动源或车辆激发点，形成多炮主动源采集。项目采用三维坐标系：x 为沿道路/沿光纤方向，y 为横穿道路方向，z 为深度向下。光纤近似位于 y = 0，震源线近似位于 y = W，地下异常体位于道路下方 x0, y0, h。

项目重点研究浅层瑞雷波在道路地下空洞、脱空、松散区、管沟、管线、弱夹层等异常体附近产生的散射、绕射、能量变化和相干性异常。通过构建道路地下层状模型、多异常体模型和 DAS-like 接收响应，生成多炮合成数据，并进一步开展绕射/散射属性提取、多炮联合扫描定位、误差与置信度评估。现阶段默认 workflow 采用快速运动学与等效散射近似，后续通过 gauge length DAS-like 近似、分层等效速度、局部 elastic2d/simplified elastic3d、FEM、SEM、BEM 和 FWI refinement 逐步提高正演和反演真实性。

本项目定位为科研级算法原型平台，不是工业级软件，也不是完整 DAS 仪器模拟或完整三维弹性波模拟系统。所有结果必须明确说明近似条件、适用范围和失败场景。

### 12.1 主入口设计

项目应提供一个清晰的本地运行入口 `main.py`。该入口不仅负责选择任务和调用 pipeline，也负责统一管理参数。项目不单独设置 `config/` 或 `para/` 文件夹，所有默认参数均在 `main.py` 的 `argparse` 中集中定义。

推荐运行方式如下：

```bash
python main.py --task full_pipeline
python main.py --task forward
python main.py --task scan
python main.py --task robustness
python main.py --task reproduce_literature
```

也可以通过命令行覆盖个别参数：

```bash
python main.py --task full_pipeline --anomaly-depth-m 4.0
python main.py --task full_pipeline --road-width-m 24.0 --source-shot-count 25
python main.py --task scan --scan-x-step-m 1.0 --scan-depth-step-m 0.25
python main.py --task robustness --noise-snr-db 5.0 --bad-channel-ratio 0.1
```

推荐任务类型包括：

```text
forward：只运行多炮 DAS-like 正演；
scan：读取或生成数据后运行多炮扫描定位；
full_pipeline：正演、特征提取、扫描定位、置信度评价和报告输出全流程；
robustness：运行噪声、坏道、速度误差和几何敏感性实验；
reproduce_literature：运行指定文献复现流程；
plot_only：只根据已有结果重新绘图；
debug：运行小规模调试流程。
```

`main.py` 的逻辑建议保持为：

```text
1. 使用 argparse 定义所有默认参数；
2. 读取命令行输入并形成 raw_args；
3. 将 raw_args 转换为统一参数对象 params；
```

### 13.4 主参数与派生参数

本项目中很多参数存在依赖关系，不能简单地把所有变量都当成独立输入参数。例如：

```text

time.dt_s 和 time.record_length_s 可以推导出 nt 和 time_axis；
das_like.gauge_length_m 和 fiber.channel_spacing_m 可以推导出 gauge_channel_count。
```

因此，项目应明确区分主参数和派生参数：

```text
主参数：用户通过 argparse 默认值或命令行参数直接设置，用于描述实验意图；
派生参数：由 resolve_derived_params() 根据主参数自动计算，不建议用户手动设置。
```

推荐规则如下：

```text
1. fiber.channel_count 优先作为主参数，fiber.x_end_m 由程序自动计算；
2. source.shot_count 优先作为主参数，source.x_end_m 由程序自动计算；
3. source.y_m 如果不显式设置，则默认等于 road.width_m；
4. scan 网格由 min、max、step 自动生成，不在算法模块中重复生成；
5. time_axis 和 nt 由 dt_s 和 record_length_s 自动生成；
6. gauge_channel_count 由 gauge_length_m 和 channel_spacing_m 自动生成；
7. output_run_dir 由 output.root_dir、run_name 和运行时间自动生成。
```

如果后续确实需要允许用户设置 `fiber.x_end_m` 或 `source.x_end_m`，则必须检查它们是否与 `channel_count`、`shot_count` 和 spacing 一致。如果不一致，应给出清楚的错误或警告，而不是静默使用其中一个。

### 13.6 参数校验

在运行任何正演、扫描或绘图之前，必须先进行参数校验。参数校验建议放在 `main.py` 中，分为两层：

```text
第一层：validate_raw_params()，检查用户直接输入的参数是否明显非法；
第二层：validate_resolved_params()，检查派生参数是否一致、维度是否合理、数组是否可用。
```

至少包括：

```text
1. 道路宽度必须大于 0；
2. 通道间距必须大于 0；
3. 炮点间距必须大于 0；
4. 采样间隔必须大于 0；
5. 记录长度必须大于采样间隔；
6. 瑞雷波速度必须大于 0；
7. 异常体深度必须大于 0；
8. 异常体 y 坐标应位于道路范围内；
9. 震源和光纤位置应与道路几何关系一致；
10. 扫描范围应覆盖合理的道路下方区域；
11. gauge length 不应小于通道间距；
12. gauge length 不应大于光纤有效长度；
13. 扫描网格点数不能过大，以免本地运行失控；
14. 噪声、坏道比例、弱耦合比例应位于合理范围；
15. 输出目录必须可创建；
16. 如果 response_level 为 point_receiver，应说明 gauge_length_m 暂不参与当前响应计算；
17. 如果 response_level 为 gauge_length_strain，则必须检查 gauge_length_m 与 channel_spacing_m 的关系。
```

如果参数不合法，程序应给出清楚的错误信息，说明是哪一个参数有问题、当前值是多少、合理范围是什么，而不是在后续算法中报难以理解的数组错误。

### 13.8 注释规范

本项目代码注释必须采用清晰、详细、教学型注释风格。尤其是几何、正演、DAS-like 响应、绕射走时、扫描定位、置信度评价等核心算法模块，应尽量做到让后续阅读者能够理解每一步的物理意义和程序逻辑。

注释要求如下：

```text
1. 每个核心函数必须说明函数用途；
2. 每个核心函数必须说明输入参数的物理含义和单位；
3. 每个核心函数必须说明输出结果的形状、单位和含义；
4. 关键公式必须在代码附近用注释解释；
5. 几何路径、走时、能量、相干性、score 等核心计算必须说明为什么这样算；
6. 数组维度必须明确说明，例如 shot × time × channel；
7. 坐标方向必须明确说明，例如 x 沿道路方向、y 横穿道路方向、z 向下为正；
8. 对近似条件必须明确说明，例如 kinematic approximation、DAS-like approximation；
9. 对容易误解的地方必须写限制说明；
10. 不允许使用过于简略、无法理解的变量名和注释。
```

推荐注释风格示例：

```python
def compute_scatter_travel_time(source_xyz, scatter_xyz, receiver_xyz, velocity_mps):
    """
    计算单个震源、单个等效散射点、单个 DAS-like 接收通道之间的散射走时。

    物理意义
    --------
    当前函数采用运动学等效散射近似。
    假设波场从震源出发，先传播到异常体等效散射点，
    再由散射点传播到 DAS-like 接收通道。
    因此总走时由两段路径组成：

        t = distance(source, scatter) / velocity
          + distance(scatter, receiver) / velocity

    注意
    ----
    这里的 velocity_mps 是等效瑞雷波速度或分层等效速度。
    本函数不模拟真实弹性波的反射、透射、模式转换和频散，
    只用于快速生成可解释的绕射/散射走时。

    参数
    ----
    source_xyz : array-like, shape = (3,)
        震源坐标，单位 m，坐标顺序为 x, y, z。
    scatter_xyz : array-like, shape = (3,)
        异常体等效散射点坐标，单位 m，坐标顺序为 x, y, z。
    receiver_xyz : array-like, shape = (3,)
        DAS-like 接收通道坐标，单位 m，坐标顺序为 x, y, z。
    velocity_mps : float
        等效传播速度，单位 m/s。

    返回
    ----
    travel_time_s : float
        source -> scatter -> receiver 的总走时，单位 s。
    """
```

核心算法内部也应保留必要的逐步注释：

```python
# 第一段路径：震源到异常体等效散射点。
# 这段路径表示入射波传播到空洞或松散区附近的时间。
dist_source_to_scatter = np.linalg.norm(scatter_xyz - source_xyz)

# 第二段路径：异常体等效散射点到 DAS-like 接收通道。
# 这段路径表示散射或绕射能量从异常体传播到光纤通道的时间。
dist_scatter_to_receiver = np.linalg.norm(receiver_xyz - scatter_xyz)

# 当前阶段采用统一等效速度，因此总走时为两段距离除以同一个速度。
# 后续如果引入分层速度模型，可以在这里替换为更复杂的走时计算器。
travel_time_s = (
    dist_source_to_scatter + dist_scatter_to_receiver
) / velocity_mps
```

### 13.9 对 Codex 的特别要求

后续使用 Codex 开发时，需要明确要求：

```text
1. 所有新增参数必须先进入 main.py 的 argparse，不允许直接写死在算法脚本中；
2. 所有模块只能读取 main.py 解析、派生和校验后的 params；
3. 某个参数变化后，相关派生参数必须自动更新；
4. 新增算法前必须检查 main.py 中是否已有对应参数分组；
5. 新增实验脚本不能重新定义一套局部参数；
6. 新增绘图脚本不能手写另一套坐标轴参数；
7. 所有核心算法必须有清晰详细的中文注释；
8. 关键函数必须说明物理意义、输入、输出、单位、数组维度和近似条件；
9. 每轮开发必须增加参数同步测试；
10. 不允许为了快速演示破坏 main.py 统一参数管理规则。
```

可以把这一条作为项目长期约束写入 README 或 `docs/workflow.md`：

```text
本项目所有流程必须以 main.py 中的 argparse 和 params 为唯一参数源。任何几何、速度、异常体、采样、DAS-like、扫描、噪声和输出参数的新增或修改，都必须先进入 main.py 的 argparse，并通过 resolve_derived_params() 和 validate_resolved_params() 进入主流程。严禁在正演、扫描、绘图、报告或实验脚本中散落定义局部参数。核心算法代码必须采用教学型中文详细注释，说明物理意义、参数单位、数组维度、近似条件和适用边界。
```
